# Simulated Expert Panel Synthesis

## Consensus

The reviewers agree that the project has strong strategic direction but must
now turn roadmap intent into executable contracts and evidence. The most common
theme is that public credibility depends on disciplined validation, source
provenance, and honest current-state language.

## Highest-Value Near-Term Work

1. Backfill roadmap governance metadata.
   This prevents inaccurate completion claims and clarifies dependencies.

2. Implement versioned reference-data and formula-bundle schemas.
   This converts roadmap intent into executable contracts.

3. Implement pricing-year validation gates.
   This prevents unsupported year or stream claims.

4. Implement SAS and Excel parity evidence.
   This is essential for mathematical and policy credibility.

5. Implement restricted artifact controls.
   This protects the repository from licensing and public-trust failures.

6. Implement one complete stream/year lifecycle.
   Use a narrow canary to prove source discovery, extraction, validation,
   Rust-core execution, Python binding, CLI output, docs, and release evidence.

7. Add public-facing appropriate-use and validation-status documentation.
   This helps users understand what is current, validated, experimental, or
   future-only.

8. Only then expand bindings and apps.
   R, Julia, C#, Go, JVM, WASM, SQL, SAS interop, and Power Platform should
   follow shared conformance tests.

## Recommended Milestone Definition

The next credible milestone should be:

- one pricing year;
- one calculator stream;
- archived and extracted SAS/Excel sources;
- source parity and output parity records;
- versioned schemas;
- Arrow-compatible batch contract;
- Rust canary or opt-in path;
- Python and CLI conformance;
- Starlight docs page;
- release evidence table.

This milestone proves the pattern before scaling the roadmap.
